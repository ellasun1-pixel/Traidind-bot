# Strategy Specification v1.0
### Paper Challenge Investment Agent — Kraken Funded Challenge

Основано на выводах завершённых этапов исследования (Feasibility Analysis, Edge Discovery, Momentum Stress Test, Regime Detector Validation, Multi-Asset Validation). Новых исследований на этом этапе не проводится.

---

## 1. Цель и границы Challenge

| Параметр | Значение |
|---|---|
| Стартовый баланс | $1000 |
| Уровень победы | $1120 (+12%) |
| Уровень поражения | $950 (−5%, фиксированно от старта) |
| Срок | не ограничен |
| Реальные сделки | НЕТ — только Paper Challenge |
| Доступ к Kraken-аккаунту | НЕ требуется, API-ключ не используется |
| Исполнение сделок | вручную пользователем, агент только сигнализирует |

**Главная целевая функция:** максимизировать P(баланс достигнет $1120 раньше, чем упадёт ниже $950).

**Не является целью:** максимизация доходности, Sharpe ratio, количества сделок.

---

## 2. Исходные принципы (из завершённых исследований)

Эти принципы обязательны и не подлежат пересмотру без нового отдельного исследовательского этапа:

1. Универсальный надёжный edge **не доказан**. Простой rule-based классификатор режима показал MCC ≈ 0.02–0.05 (около случайного угадывания).
2. Momentum **не является** постоянно работающим ядром стратегии — в реалистичной смеси режимов рынка слепой momentum статистически неотличим от отсутствия edge, а в периоды консолидации может быть хуже.
3. Рынок **нельзя считать надёжно предсказуемым** на публичных данных доступными агенту средствами.
4. TREND-фильтр используется **только как защитный и контекстный механизм** (снижает вероятность поражения), а не как источник дополнительной доходности.
5. По умолчанию агент **консервативен**. При неуверенности — решение `NO TRADE`.
6. Риск-менеджмент не создаёт edge — он определяет, насколько эффективно уже существующий (ограниченный) edge конвертируется в вероятность прохождения.
7. Задержка ручного исполнения существенно снижает эффективность любого сигнала — стратегия должна быть устойчива к задержке 1–3 дня/часа, а не оптимизирована под мгновенное исполнение.

## 3. Активы

Временный базовый список (конфигурируемый, заменяется при получении точного списка Kraken Funded):

```
BTC/USD, ETH/USD, XRP/USD, LINK/USD, LTC/USD
```

Правила:
- Не торговать мелкими и малоликвидными монетами.
- Состав не зависит от личного портфеля пользователя.
- Список задаётся конфигурацией (`config/assets.yaml` или аналог), не хардкодится.

## 4. Режимы рынка (регистрируются, но используются только как контекст)

Причинные (без look-ahead) определения, идентичные валидированным ранее:

```
PANIC:  падение ≥10% за 48ч  И  реализованная волатильность > 1.8× медианной за 252 дня
LOWVOL: волатильность ≤ 25-й процентиль (252 дня)  И  ER20 < 0.35
TREND:  ER20 ≥ 0.35  И  цена > EMA200  И  EMA50 > EMA200
CHOP:   всё остальное (по умолчанию)
```

Дополнительно вычисляется индикатор силы тренда (ADX(14), если доступны OHLC; иначе Kaufman Efficiency Ratio как заявленный, менее надёжный прокси — использовать с пометкой пониженной достоверности).

**Важно:** классификация режима — вспомогательный, не самодостаточный сигнал. Она не должна использоваться как единственное основание для BUY.

## 5. Правила риск-менеджмента

### 5.1 Жёсткие ограничения (не настраиваются пользователем без явного подтверждения)

```
NO_LEVERAGE = true
NO_SHORTS = true
NO_FUTURES = true
NO_AVERAGING_DOWN = true
MAX_OPEN_POSITIONS = 2
```

### 5.2 Риск на сделку и общий риск

```
RISK_PER_TRADE_PCT = 1.5%–2.2% стартового баланса     (≈ $15.00–$22.00)
RISK_PER_TRADE_PCT_DEFAULT = 1.8%                      (≈ $18.00)
MAX_TOTAL_OPEN_RISK_PCT = 4.0% стартового баланса     (≈ $40.00)
```

### 5.3 Расчёт размера позиции

Позиция рассчитывается через допустимый убыток, а не через желаемую сумму вложения:

```
risk_dollars = выбранный риск сделки (в границах 5.2)
stop_distance_pct = |entry_price − stop_loss_price| / entry_price
position_value_usd = risk_dollars / stop_distance_pct
```

После расчёта — проверка ограничений портфеля (5.1, 5.2, 5.4) и уменьшение при конфликте.

### 5.4 Экспозиция по уровням баланса (циркуит-брейкеры)

| Условие баланса | Правило |
|---|---|
| баланс < $1050 | минимум 50% капитала в USD |
| баланс ≥ $1050 | защищать полученную прибыль, снижать общий риск |
| баланс ≥ $1090 | новые сделки только при очень сильном сигнале; минимум 80% в USD |
| баланс ≥ $1110 | новые обычные позиции не открываются; цель — сохранить баланс до $1120 |
| баланс < $975 | новые покупки запрещены; разрешены только сокращение риска и выход |
| баланс < $965 | рекомендация `MOVE TO USD` |
| баланс ≤ $955 | постоянное критическое предупреждение; никаких новых сделок |

## 6. Условия подачи сигналов

### 6.1 BUY разрешён только если ВСЕ условия выполнены:
- режим рынка = TREND (BUY запрещён в CHOP, LOWVOL, PANIC);
- актив достаточно ликвиден (входит в список раздела 3);
- долгосрочный тренд (EMA200 на дневном таймфрейме) не отрицательный;
- сигнал подтверждён закрытой свечой (дневной или 4-часовой), не текущей незакрытой;
- ожидаемая прибыль существенно выше суммы комиссии + спреда + допустимой ошибки;
- сделка не нарушает риск-бюджет (5.2–5.4);
- сигнал остаётся актуальным в момент фактической отправки уведомления (повторная проверка перед отправкой);
- **не** отправлять BUY сразу после резкого вертикального роста (>8% за короткий период — раздел 8).

### 6.2 SELL / REDUCE отправляется, если хотя бы одно условие верно:
- нарушен защитный уровень (STOP-LOSS);
- фундаментально изменился режим рынка (например, вход в PANIC);
- позиция стала непропорционально большой относительно портфеля;
- баланс приблизился к уровню поражения (раздел 5.4);
- ожидаемый дальнейший риск выше ожидаемой выгоды.

### 6.3 TAKE PROFIT
Отправляется только при существенной прибыли по позиции (порог конфигурируется, по умолчанию — прибыль, кратно превышающая риск сделки). Не уведомлять ради фиксации нескольких центов.

### 6.4 NO TRADE
Полноценное, часто правильное решение. Не создаёт уведомления пользователю (см. раздел 9).

## 7. Таймфреймы и частота анализа

```
Проверка рыночных данных:      каждые 15 минут
Основные торговые решения:     дневные свечи + 4-часовые свечи
1-часовые свечи:                только для проверки срочности/качества исполнения
```

Стратегия **не** основана на минутных колебаниях или скальпинге.

---

## 8. Пороги значимого движения (конфигурируемые)

```
< 1%             — как правило, игнорировать
1–3%             — проанализировать, уведомление не обязательно
> 3% за короткий период  — переоценить ситуацию
> 5%             — потенциально важное движение
> 8%             — экстренная переоценка риска
```

Сам процент **не создаёт сделку автоматически** — он только запускает повторный анализ существующего плана.

## 9. Логика уведомлений

### 9.1 Активные часы
`Asia/Jerusalem`, 08:00–23:00 — обычные торговые сигналы разрешены.

### 9.2 Ночной режим (23:00–08:00)
- Обычные BUY-сигналы **не отправляются**.
- Агент продолжает наблюдение и логирование всех событий.
- Утром (в 08:00) — проверка актуальности ночных сигналов; если устарел — сообщить, что действие не требуется.
- Экстренное уведомление ночью разрешено **только** при: угрозе пробития защитного уровня открытой позиции, движении портфеля к $950, обвале крупного актива, событии, требующем защиты открытой позиции.

### 9.3 Что создаёт уведомление (единственные основания)
1. Нужно открыть позицию.
2. Нужно закрыть позицию.
3. Нужно уменьшить позицию.
4. Нужно зафиксировать существенную прибыль.
5. Нужно защититься от существенного падения.
6. Резкое изменение рынка меняет предыдущий план.
7. Баланс приближается к уровню поражения.
8. Баланс приближается к цели.
9. Требуется срочное действие.

Во всех остальных случаях — `NO TRADE` без уведомления.

### 9.4 Приоритеты уведомлений
`CRITICAL` / `HIGH` / `MEDIUM`

## 10. Формат сигнала (обязательные поля)

1. Приоритет.
2. Решение (`BUY` / `SELL` / `HOLD` / `REDUCE` / `TAKE PROFIT` / `MOVE TO USD`).
3. Актив.
4. Точная сумма в USD.
5. Допустимый диапазон цены исполнения.
6. Тип ордера (`LIMIT` / `MARKET`).
7. Уровень отмены сигнала.
8. Защитный уровень выхода (`STOP-LOSS`).
9. Максимальный возможный убыток в USD.
10. Объяснение простым языком.
11. Причина решения.
12. Остаток USD после сделки.
13. Текущий виртуальный баланс.
14. Расстояние до $1120.
15. Расстояние до $950.

## 12. Paper Challenge — учёт

Виртуальный счёт ведёт учёт:
- цены исполнения, комиссии, спреда;
- количества актива и средней цены;
- реализованной и нереализованной прибыли (`P&L`, `Unrealized P&L`);
- текущего баланса, `Drawdown`;
- расстояния до цели и до поражения.

**Сделка засчитывается только после ручного подтверждения пользователем** (`/confirm`). Неподтверждённые сигналы не отражаются как исполненные в виртуальном портфеле.

---

## 13. Источники данных

- Только публичные рыночные данные (публичный Kraken API, публичный Coinbase API, CoinGecko и т.п.).
- Приватный Kraken API и ключи — не используются и не хранятся.
- Обязателен резервный источник данных.
- Если источники расходятся более чем на заданный процент (конфигурируемый порог) — сигнал не выдаётся.

---

## 14. Режимы работы агента

| Режим | Поведение |
|---|---|
| `PAPER_CHALLENGE` | основной режим — анализ + виртуальные сделки + уведомления |
| `ALERT_ONLY` | только анализ и уведомления, без ведения виртуального портфеля |
| `REPLAY` | воспроизведение исторического периода для проверки логики |
| `PAUSED` | наблюдение продолжается, торговые сигналы не выдаются |

Автоматическое исполнение сделок отсутствует во всех режимах.

---

## 15. Критерии соответствия спецификации

Реализация считается соответствующей этой спецификации, если:
- ни при каких условиях не выдаётся BUY при нарушении раздела 6.1;
- размер позиции всегда получен из формулы раздела 5.3, а не назначен произвольно;
- ни один из циркуит-брейкеров раздела 5.4 не может быть проигнорирован;
- в ночное время (23:00–08:00) не отправляется ни одного обычного BUY-уведомления;
- решение `NO TRADE` не создаёт сообщение пользователю;
- каждое реальное уведомление содержит все 15 полей раздела 10;
- без ручного `/confirm` сделка не отражается как исполненная.

Эта спецификация является основанием для Technical Specification v1.0 и последующей реализации кода.
